Détail des vulnérabilités critiques / hautes

V01 / V02 CRITICAL NoOpPasswordEncoder — mots de passe non hashés +
Description

Le provider de mots de passe utilise NoOpPasswordEncoder — les mots de passe sont stockés et comparés en clair. En cas de dump de base de données ou de fuite de logs, tous les comptes sont immédiatement compromis.

Preuve
@Bean
public NoOpPasswordEncoder passwordEncoder() {
  return (NoOpPasswordEncoder) NoOpPasswordEncoder.getInstance();
}
Localisations
container/WebSecurityConfig.java#L87

webwolf/WebSecurityConfig.java#L85
Recommandations
  • Remplacer par BCryptPasswordEncoder ou Argon2PasswordEncoder
  • Utiliser PasswordEncoderFactories.createDelegatingPasswordEncoder()
  • Migrer les hashes existants (prefixe {bcrypt})
V03 / V04 / V05 HIGH Mauvaise configuration d'erreurs et endpoints admin +
Description

Stacktrace forcée en réponse HTTP (include-stacktrace=always), endpoints Actuator sensibles exposés, et règle de sécurité qui autorise /actuator/** sans authentification. Surface d'attaque élargie + fuite de configuration interne.

Localisations
application-webgoat.properties#L1

application-webgoat.properties#L68

container/WebSecurityConfig.java#L43
Recommandations
  • Passer server.error.include-stacktrace=never
  • Restreindre : management.endpoints.web.exposure.include=health,info
  • Supprimer permitAll sur /actuator/** et protéger par rôle admin
V06 HIGH Injection SQL — DDL construit par concaténation +
Description

Construction dynamique d'une requête SQL DDL avec le username utilisateur sans échappement. Si la validation d'entrée est contournée, un attaquant peut injecter du SQL arbitraire.

Preuve
jdbcTemplate.execute(
  "CREATE SCHEMA \"" + webGoatUser.getUsername() + "\" authorization dba"
);
Localisation
container/users/UserService.java#L53
Recommandations
  • Validation whitelist stricte : ^[a-zA-Z0-9_]{1,32}$
  • Éviter la concaténation SQL dans les DDL
  • Encapsuler et échapper via API dédiée
V07 HIGH Path traversal — upload de fichier +
Description

Le nom de fichier fourni par l'utilisateur est résolu sans normalisation ni contrôle de préfixe. Un payload contenant ../ peut permettre d'écrire hors du répertoire cible.

Preuve
var destinationFile = destinationDir.toPath()
    .resolve(multipartFile.getOriginalFilename());
Files.copy(is, destinationFile); // ⚠ pas de contrôle
Localisation
webwolf/FileServer.java#L75
Recommandations
  • Appeler .normalize() sur le chemin résolu
  • Vérifier que le chemin commence bien par le répertoire base
  • Refuser les noms contenant .., séparateurs ou chemins absolus
